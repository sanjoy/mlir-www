31 /// The Simplex class implements a version of the Simplex and Generalized Basis
32 /// Reduction algorithms, which can perform analysis of integer sets with affine
33 /// inequalities and equalities. A Simplex can be constructed
34 /// by specifying the dimensionality of the set. It supports adding affine
35 /// inequalities and equalities, and can perform emptiness checks, i.e., it can
36 /// find a solution to the set of constraints if one exists, or say that the
37 /// set is empty if no solution exists. Currently, this only works for bounded
38 /// sets. Furthermore, it can find a subset of these constraints that are
39 /// redundant, i.e. a subset of constraints that doesn't constrain the affine
40 /// set further after adding the non-redundant constraints. Simplex can also be
41 /// constructed from an IntegerPolyhedron object.
42 ///
43 /// The implementation of the Simplex and SimplexBase classes, other than the
44 /// functionality for sampling, is based on the paper
45 /// "Simplify: A Theorem Prover for Program Checking"
46 /// by D. Detlefs, G. Nelson, J. B. Saxe.
47 ///
48 /// We define variables, constraints, and unknowns. Consider the example of a
49 /// two-dimensional set defined by 1 + 2x + 3y >= 0 and 2x - 3y >= 0. Here,
50 /// x, y, are variables while 1 + 2x + 3y >= 0, 2x - 3y >= 0 are
51 /// constraints. Unknowns are either variables or constraints, i.e., x, y,
52 /// 1 + 2x + 3y >= 0, 2x - 3y >= 0 are all unknowns.
53 ///
54 /// The implementation involves a matrix called a tableau, which can be thought
55 /// of as a 2D matrix of rational numbers having number of rows equal to the
56 /// number of constraints and number of columns equal to one plus the number of
57 /// variables. In our implementation, instead of storing rational numbers, we
58 /// store a common denominator for each row, so it is in fact a matrix of
59 /// integers with number of rows equal to number of constraints and number of
60 /// columns equal to _two_ plus the number of variables. For example, instead of
61 /// storing a row of three rationals [1/2, 2/3, 3], we would store [6, 3, 4, 18]
62 /// since 3/6 = 1/2, 4/6 = 2/3, and 18/6 = 3.
63 ///
64 /// Every row and column except the first and second columns is associated with
65 /// an unknown and every unknown is associated with a row or column. The second
66 /// column represents the constant, explained in more detail below. An unknown
67 /// associated with a row or column is said to be in row or column position
68 /// respectively.
69 ///
70 /// The vectors var and con store information about the variables and
71 /// constraints respectively, namely, whether they are in row or column
72 /// position, which row or column they are associated with, and whether they
73 /// correspond to a variable or a constraint.
74 ///
75 /// An unknown is addressed by its index. If the index i is non-negative, then
76 /// the variable var[i] is being addressed. If the index i is negative, then
77 /// the constraint con[~i] is being addressed. Effectively this maps
78 /// 0 -> var[0], 1 -> var[1], -1 -> con[0], -2 -> con[1], etc. rowUnknown[r] and
79 /// colUnknown[c] are the indexes of the unknowns associated with row r and
80 /// column c, respectively.
81 ///
82 /// The unknowns in column position are together called the basis. Initially the
83 /// basis is the set of variables -- in our example above, the initial basis is
84 /// x, y.
85 ///
86 /// The unknowns in row position are represented in terms of the basis unknowns.
87 /// If the basis unknowns are u_1, u_2, ... u_m, and a row in the tableau is
88 /// d, c, a_1, a_2, ... a_m, this represents the unknown for that row as
89 /// (c + a_1*u_1 + a_2*u_2 + ... + a_m*u_m)/d. In our running example, if the
90 /// basis is the initial basis of x, y, then the constraint 1 + 2x + 3y >= 0
91 /// would be represented by the row [1, 1, 2, 3].
92 ///
93 /// The association of unknowns to rows and columns can be changed by a process
94 /// called pivoting, where a row unknown and a column unknown exchange places
95 /// and the remaining row variables' representation is changed accordingly
96 /// by eliminating the old column unknown in favour of the new column unknown.
97 /// If we had pivoted the column for x with the row for 2x - 3y >= 0,
98 /// the new row for x would be [2, 1, 3] since x = (1*(2x - 3y) + 3*y)/2.
99 /// See the documentation for the pivot member function for details.
100 ///
101 /// The association of unknowns to rows and columns is called the _tableau
102 /// configuration_. The _sample value_ of an unknown in a particular tableau
103 /// configuration is its value if all the column unknowns were set to zero.
104 /// Concretely, for unknowns in column position the sample value is zero and
105 /// for unknowns in row position the sample value is the constant term divided
106 /// by the common denominator.
107 ///
108 /// The tableau configuration is called _consistent_ if the sample value of all
109 /// restricted unknowns is non-negative. Initially there are no constraints, and
110 /// the tableau is consistent. When a new constraint is added, its sample value
111 /// in the current tableau configuration may be negative. In that case, we try
112 /// to find a series of pivots to bring us to a consistent tableau
113 /// configuration, i.e. we try to make the new constraint's sample value
114 /// non-negative without making that of any other constraints negative. (See
115 /// findPivot and findPivotRow for details.) If this is not possible, then the
116 /// set of constraints is mutually contradictory and the tableau is marked
117 /// _empty_, which means the set of constraints has no solution.
118 ///
119 /// The Simplex class supports redundancy checking via detectRedundant and
120 /// isMarkedRedundant. A redundant constraint is one which is never violated as
121 /// long as the other constraints are not violated, i.e., removing a redundant
122 /// constraint does not change the set of solutions to the constraints. As a
123 /// heuristic, constraints that have been marked redundant can be ignored for
124 /// most operations. Therefore, these constraints are kept in rows 0 to
125 /// nRedundant - 1, where nRedundant is a member variable that tracks the number
126 /// of constraints that have been marked redundant.
127 ///
128 /// This Simplex class also supports taking snapshots of the current state
129 /// and rolling back to prior snapshots. This works by maintaining an undo log
130 /// of operations. Snapshots are just pointers to a particular location in the
131 /// log, and rolling back to a snapshot is done by reverting each log entry's
132 /// operation from the end until we reach the snapshot's location.
133 ///
134 /// Finding an integer sample is done with the Generalized Basis Reduction
135 /// algorithm. See the documentation for findIntegerSample and reduceBasis.
136 ///
137 /// The SimplexBase class contains some basic functionality. In the future,
138 /// lexicographic optimization will be supported by a class inheriting from
139 /// SimplexBase. Functionality that will not supported by that class is placed
140 /// in `Simplex`.

200  /// An Unknown is either a variable or a constraint. It is always associated
201  /// with either a row or column. Whether it's a row or a column is specified
202  /// by the orientation and pos identifies the specific row or column it is
203  /// associated with. If the unknown is restricted, then it has a
204  /// non-negativity constraint associated with it, i.e., its sample value must
205  /// always be non-negative and if it cannot be made non-negative without
206  /// violating other constraints, the tableau is empty.
207  struct Unknown {
208  Unknown(Orientation oOrientation, bool oRestricted, unsigned oPos)
209  : pos(oPos), orientation(oOrientation), restricted(oRestricted) {}
210  unsigned pos;
211  Orientation orientation;
212  bool restricted : 1;
213 
214  void print(raw_ostream &os) const {
215  os << (orientation == Orientation::Row ? "r" : "c");
216  os << pos;
217  if (restricted)
218  os << " [>=0]";
219  }

335  /// Compute the maximum or minimum value of the given row, depending on
336  /// direction. The specified row is never pivoted. On return, the row may
337  /// have a negative sample value if the direction is down.
338  ///
339  /// Returns a Fraction denoting the optimum, or a null value if no optimum
340  /// exists, i.e., if the expression is unbounded in this direction.
341  Optional<Fraction> computeRowOptimum(Direction direction, unsigned row);
342 
343  /// Compute the maximum or minimum value of the given expression, depending on
344  /// direction. Should not be called when the Simplex is empty.
345  ///
346  /// Returns a Fraction denoting the optimum, or a null value if no optimum
347  /// exists, i.e., if the expression is unbounded in this direction.
348  Optional<Fraction> computeOptimum(Direction direction,
349  ArrayRef<int64_t> coeffs);
350 
351  /// Returns whether the perpendicular of the specified constraint is a
352  /// is a direction along which the polytope is bounded.
353  bool isBoundedAlongConstraint(unsigned constraintIndex);
354 
355  /// Returns whether the specified constraint has been marked as redundant.
356  /// Constraints are numbered from 0 starting at the first added inequality.
357  /// Equalities are added as a pair of inequalities and so correspond to two
358  /// inequalities with successive indices.
359  bool isMarkedRedundant(unsigned constraintIndex) const;
360 
361  /// Finds a subset of constraints that is redundant, i.e., such that
362  /// the set of solutions does not change if these constraints are removed.
363  /// Marks these constraints as redundant. Whether a specific constraint has
364  /// been marked redundant can be queried using isMarkedRedundant.
365  void detectRedundant();
366 
367  /// Returns a (min, max) pair denoting the minimum and maximum integer values
368  /// of the given expression.
369  std::pair<int64_t, int64_t> computeIntegerBounds(ArrayRef<int64_t> coeffs);
370 
371  /// Returns true if the polytope is unbounded, i.e., extends to infinity in
372  /// some direction. Otherwise, returns false.
373  bool isUnbounded();
374 
375  /// Make a tableau to represent a pair of points in the given tableaus, one in
376  /// tableau A and one in B.
377  static Simplex makeProduct(const Simplex &a, const Simplex &b);
378 
379  /// Returns an integer sample point if one exists, or None
380  /// otherwise. This should only be called for bounded sets.
381  Optional<SmallVector<int64_t, 8>> findIntegerSample();
382 
383  /// Check if the specified inequality already holds in the polytope.
384  bool isRedundantInequality(ArrayRef<int64_t> coeffs);
385 
386  /// Check if the specified equality already holds in the polytope.
387  bool isRedundantEquality(ArrayRef<int64_t> coeffs);
388 
389  /// Returns true if this Simplex's polytope is a rational subset of `poly`.
390  /// Otherwise, returns false.
391  bool isRationalSubsetOf(const IntegerPolyhedron &poly);
392 
393 private:
394  friend class GBRSimplex;
395 
396  /// Compute the maximum or minimum of the specified Unknown, depending on
397  /// direction. The specified unknown may be pivoted. If the unknown is
398  /// restricted, it will have a non-negative sample value on return.
399  /// Should not be called if the Simplex is empty.
400  ///
401  /// Returns a Fraction denoting the optimum, or a null value if no optimum
402  /// exists, i.e., if the expression is unbounded in this direction.
403  Optional<Fraction> computeOptimum(Direction direction, Unknown &u);
404 
405  /// Mark the specified unknown redundant. This operation is added to the undo
406  /// log and will be undone by rollbacks. The specified unknown must be in row
407  /// orientation.
408  void markRowRedundant(Unknown &u);
409 
410  /// Reduce the given basis, starting at the specified level, using general
411  /// basis reduction.
412  void reduceBasis(Matrix &basis, unsigned level);
